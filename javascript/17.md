## 17. 생성자 함수에 의한 객체 생성

객체 리터럴은 간단하지만, 같은 구조의 객체를 많이 만들어야 할 때는 비효율적이다.  
이 장은 생성자 함수를 이용해 인스턴스를 반복 생성하는 방법과 `new` 동작 원리를 다룬다.

## 17.1 Object 생성자 함수

- `new Object()`로 빈 객체를 만든 뒤 프로퍼티/메서드를 추가할 수 있다.
- 하지만 보통은 객체 리터럴 `{}`이 더 간단하므로 실무에서 `new Object()`는 드물다.

```js
const person = new Object();
person.name = "Lee";
person.sayHello = function () {
  console.log("Hi! My name is " + this.name);
};
```

## 17.2 생성자 함수

### 17.2.1 객체 리터럴 방식의 문제점

- 동일한 구조의 객체를 여러 개 만들 때, 같은 메서드를 매번 중복 작성하게 된다.
- 상태(프로퍼티 값)는 다를 수 있지만, 동작(메서드)은 보통 동일하다.

### 17.2.2 생성자 함수 방식의 장점

- 공통 구조를 템플릿처럼 정의하고 인스턴스를 쉽게 여러 개 생성할 수 있다.

```js
function Circle(radius) {
  this.radius = radius;
  this.getDiameter = function () {
    return 2 * this.radius;
  };
}

const circle1 = new Circle(5);
const circle2 = new Circle(10);
```

### this 요약

- 일반 함수 호출: `this` -> 전역 객체(엄격 모드에서는 `undefined`)
- 메서드 호출: `this` -> 메서드를 호출한 객체
- 생성자 함수 호출(`new`): `this` -> 생성될 인스턴스

### 17.2.3 생성자 함수의 인스턴스 생성 과정

`new Circle(5)` 실행 시 엔진은 암묵적으로 다음을 수행한다.

1. 빈 객체(인스턴스) 생성
2. `this`를 그 인스턴스에 바인딩
3. 함수 본문 실행으로 인스턴스 초기화
4. `this`를 암묵적으로 반환

주의:
- 생성자에서 객체를 `return`하면 그 객체가 반환된다.
- 원시값을 `return`하면 무시되고 `this`가 반환된다.

### 17.2.4 [[Call]]과 [[Construct]]

- 함수는 호출 가능한 객체다.
- 일반 호출 시 내부 메서드 `[[Call]]` 실행
- `new`와 함께 호출 시 `[[Construct]]` 실행

### 17.2.5 constructor vs non-constructor

- constructor: 함수 선언문, 함수 표현식
- non-constructor: 화살표 함수, ES6 메서드 축약 표현, 클래스 메서드
- non-constructor에 `new`를 붙이면 `TypeError`

```js
const arrow = () => {};
// new arrow(); // TypeError

const obj = {
  x() {},
};
// new obj.x(); // TypeError
```

### 17.2.6 new 연산자

- `new`와 함께 호출하면 생성자 함수처럼 동작한다(단, constructor여야 함).
- `new` 없이 호출하면 일반 함수 호출이며 `[[Call]]`이 실행된다.

```js
function add(x, y) {
  return x + y;
}

const inst = new add(); // {}
```

### 17.2.7 new.target

- 함수가 `new`와 함께 호출됐는지 확인하는 메타 프로퍼티
- `new` 없이 호출되면 `new.target === undefined`
- 안전하게 생성자 호출을 강제할 수 있다.

```js
function Circle(radius) {
  if (!new.target) {
    return new Circle(radius);
  }
  this.radius = radius;
}
```

구형 환경에서는 `instanceof`를 활용한 스코프 세이프 생성자 패턴을 사용한다.

## 빌트인 생성자 호출 차이

- `Object`, `Function`: `new` 없이 호출해도 유사하게 동작
- `String`, `Number`, `Boolean`:
  - `new`와 함께 호출 -> 래퍼 객체 생성
  - `new` 없이 호출 -> 원시값 반환(명시적 타입 변환에 활용)

```js
console.log(String(123), typeof String(123)); // "123", "string"
console.log(Number("123"), typeof Number("123")); // 123, "number"
console.log(Boolean("true"), typeof Boolean("true")); // true, "boolean"
```

## 핵심 정리

- 동일 구조의 객체를 반복 생성할 때 생성자 함수가 유리하다.
- 생성자 호출의 본질은 `new` + `[[Construct]]` 실행이다.
- `new`를 빼먹으면 일반 함수 호출이 되어 의도와 다른 동작이 발생할 수 있다.
- `new.target`은 이런 실수를 방지하는 안전장치다.
- 화살표 함수/메서드 축약 표현은 생성자로 쓸 수 없다.

## 의문점 제기(생각해볼 질문)

- 생성자 함수 내부에서 메서드를 직접 정의하면 인스턴스마다 함수가 복제되는데, 언제 문제가 될까?
- 메서드는 왜 보통 프로토타입에 올리는 것이 메모리와 성능 면에서 유리할까?
- `class` 문법은 생성자 함수 패턴을 얼마나 추상화해 주는가?
- `new` 없이도 동작하게 만드는 패턴은 편의성일까, 오용을 숨기는 안티패턴일까?
- 빌트인 생성자 중 `Object/Function`과 `String/Number/Boolean`의 `new` 동작 차이는 왜 생겼을까?
- 함수 표현식은 constructor인데 ES6 메서드 축약 표현은 non-constructor인 이유는 무엇일까?

## 의문점 답변

- 인스턴스 수가 많거나 자주 생성/폐기되는 경우, 생성자 내부 메서드 복제는 메모리 사용량과 GC 비용 증가로 바로 문제가 된다.
- 프로토타입 메서드는 함수 객체를 1개만 만들고 모든 인스턴스가 공유하므로 메모리 절약과 생성 속도 측면에서 유리하다.
- `class`는 프로토타입 연결, `constructor`, 상속 문법을 일관된 형태로 감싸 주고, 실수 가능성을 줄인다.
- `new` 없이도 동작하게 하는 패턴은 라이브러리 DX에는 유리하지만, 호출 의도를 흐리면 안티패턴이 된다. 팀 규칙으로 일관성 있게 써야 한다.
- `Object/Function`은 팩토리처럼 동작해도 무리 없지만, `String/Number/Boolean`은 원시값 변환과 래퍼 객체 생성이라는 두 역할을 분리해 역사적으로 유지된 차이다.
- ES6 메서드 축약 표현은 "객체의 메서드" 의미를 강화하기 위해 생성자 호출을 금지했다. 반면 함수 선언/표현식은 독립 함수로서 constructor 자격을 유지한다.
