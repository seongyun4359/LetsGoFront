## 14. 전역 변수의 문제점

전역 변수의 무분별한 사용은 위험하다. 전역 변수를 반드시 사용해야 할 이유가 없다면 지역 변수를 사용하는 것이 좋다.

## 14.1 변수의 생명 주기

변수는 선언으로 생성되고, 할당으로 값을 가지며, 언젠가 소멸한다. 즉 변수에도 생명 주기가 있다.

### 14.1.1 지역 변수의 생명 주기

함수 내부에서 선언된 지역 변수는 함수가 호출될 때 생성되고, 함수가 종료되면 보통 소멸한다.

```js
function foo() {
  var x = "local";
  console.log(x); // local
  return x;
}

foo();

// console.log(x); // ReferenceError
```

지역 변수 `x`는 `foo` 함수가 호출되기 전에는 생성되지 않는다. 함수가 호출되면 함수 몸체 코드가 실행되기 전에 지역 변수 선언이 먼저 처리되고 `undefined`로 초기화된다.

```js
function foo() {
  var x = "local";
  console.log(x);
  return x;
}
```

지역 변수의 생명 주기는 대체로 함수의 생명 주기와 일치한다. 다만 클로저처럼 누군가 함수의 스코프를 계속 참조하면 함수가 종료되어도 지역 변수가 살아남을 수 있다.

지역 변수 호이스팅은 함수 스코프 단위로 동작한다.

```js
var x = "global";

function foo() {
  console.log(x); // undefined
  var x = "local";
}

foo();
console.log(x); // global
```

`foo` 함수 내부의 `var x`는 함수 전체에서 유효하다. 따라서 `console.log(x)`는 전역 변수 `x`가 아니라 지역 변수 `x`를 참조하고, 아직 할당 전이므로 `undefined`가 출력된다.

### 14.1.2 전역 변수의 생명 주기

전역 코드는 함수처럼 명시적으로 호출하지 않아도 코드가 로드되면 바로 실행된다.

`var` 키워드로 선언한 전역 변수는 전역 객체의 프로퍼티가 된다.

- 브라우저 환경: 전역 객체는 `window`
- Node.js 환경: 전역 객체는 `global`
- ES2020 이후: `globalThis`로 통일된 접근 가능

브라우저에서 `var`로 선언한 전역 변수는 `window`의 프로퍼티가 되므로, 웹페이지를 닫기 전까지 유지된다.

```js
var x = "global";

console.log(window.x); // global (브라우저 환경)
```

즉, 전역 변수의 생명 주기는 전역 객체의 생명 주기와 같다.

## 14.2 전역 변수의 문제점

### 암묵적 결합

전역 변수는 코드 어디서든 참조하고 변경할 수 있다. 이는 모든 코드가 전역 변수에 암묵적으로 결합된다는 뜻이다.

```js
var count = 0;

function increase() {
  count++;
}
```

전역 변수의 유효 범위가 넓을수록 누가 언제 값을 바꾸는지 추적하기 어려워지고, 의도치 않은 상태 변경 위험이 커진다.

### 긴 생명 주기

전역 변수는 애플리케이션이 종료될 때까지 살아 있을 수 있다. 따라서 메모리를 오래 점유하고, 상태가 변경될 수 있는 시간도 길다.

`var`는 중복 선언을 허용하므로 전역 변수 이름이 중복되면 의도치 않은 재할당이 발생할 수 있다.

```js
var x = 1;

var x = 100;

console.log(x); // 100
```

### 스코프 체인의 종점에 존재

전역 변수는 스코프 체인의 가장 마지막에 있다. 변수를 검색할 때 현재 스코프부터 상위 스코프로 올라가며 찾으므로, 전역 변수는 가장 마지막에 검색된다.

성능 차이가 크지는 않더라도, 구조적으로 전역 변수 검색은 가장 늦게 일어난다.

### 네임스페이스 오염

자바스크립트는 파일이 나뉘어 있어도 하나의 전역 스코프를 공유할 수 있다.

따라서 서로 다른 파일에서 같은 이름의 전역 변수나 전역 함수를 선언하면 충돌이 발생할 수 있다.

```js
// file1.js
var user = "Lee";

// file2.js
var user = "Kim";
```

이 경우 같은 전역 스코프에서 `user`가 중복되어 예상치 못한 결과가 생길 수 있다.

## 14.3 전역 변수의 사용을 억제하는 방법

전역 변수를 완전히 금지하라는 뜻은 아니다. 다만 반드시 필요한 경우가 아니라면 변수의 스코프는 최대한 좁게 유지하는 것이 좋다.

### 14.3.1 즉시 실행 함수

즉시 실행 함수(IIFE)로 코드를 감싸면 변수들이 함수의 지역 변수가 된다.

```js
(function () {
  var foo = 10;
  // ...
})();

// console.log(foo); // ReferenceError
```

이 방식은 전역 변수를 만들지 않으므로 라이브러리 코드에서 자주 사용되었다.

### 14.3.2 네임스페이스 객체

전역에 하나의 객체를 만들고, 전역 변수처럼 사용하고 싶은 값을 그 객체의 프로퍼티로 추가하는 방식이다.

```js
var MYAPP = {};

MYAPP.name = "Lee";

console.log(MYAPP.name); // Lee
```

계층적으로 구성할 수도 있다.

```js
var MYAPP = {};

MYAPP.person = {
  name: "Lee",
  address: "Seoul"
};

console.log(MYAPP.person.name); // Lee
```

단점은 네임스페이스 객체 자체가 여전히 전역 변수라는 점이다. 식별자 충돌을 줄이는 효과는 있지만 전역 오염을 완전히 해결하지는 못한다.

### 14.3.3 모듈 패턴

모듈 패턴은 관련 있는 변수와 함수를 즉시 실행 함수로 감싸 하나의 모듈처럼 만드는 방식이다.

이 방식은 전역 변수 억제뿐 아니라 캡슐화와 정보 은닉을 어느 정도 구현할 수 있다.

```js
var Counter = (function () {
  // private 변수
  var num = 0;

  // 외부에 공개할 메서드만 반환
  return {
    increase() {
      return ++num;
    },
    decrease() {
      return --num;
    }
  };
})();

console.log(Counter.num);        // undefined
console.log(Counter.increase()); // 1
console.log(Counter.increase()); // 2
console.log(Counter.decrease()); // 1
```

반환된 객체의 프로퍼티는 외부에 공개되는 퍼블릭 멤버다. 반환하지 않은 변수나 함수는 외부에서 접근할 수 없는 프라이빗 멤버처럼 동작한다.

이 구조는 클로저를 기반으로 한다.

### 14.3.4 ES6 모듈

ES6 모듈을 사용하면 파일 자체의 독자적인 모듈 스코프가 만들어진다.

따라서 모듈 안에서 `var`로 선언한 변수도 전역 변수가 아니며, `window` 객체의 프로퍼티도 아니다.

```html
<script type="module" src="lib.mjs"></script>
<script type="module" src="app.mjs"></script>
```

모던 브라우저에서는 `type="module"`을 사용해 ES6 모듈을 사용할 수 있다. 다만 실제 프로젝트에서는 브라우저 호환성, 트랜스파일링, 번들링 문제 때문에 Webpack, Vite 같은 번들러를 함께 사용하는 경우가 많다.

---

## 내용 정리

- 변수는 생성되고 소멸되는 생명 주기를 가진다.
- 지역 변수는 함수 호출 시 생성되고 함수 종료 시 보통 소멸한다.
- 전역 변수는 전역 객체의 프로퍼티가 되며 전역 객체의 생명 주기와 함께한다.
- 전역 변수는 암묵적 결합, 긴 생명 주기, 검색 비용, 네임스페이스 오염 문제를 만든다.
- 전역 변수 사용을 줄이려면 즉시 실행 함수, 네임스페이스 객체, 모듈 패턴, ES6 모듈을 사용할 수 있다.
- 네임스페이스 객체는 충돌을 줄이지만 객체 자체가 전역이라는 한계가 있다.
- 모듈 패턴은 클로저를 이용해 정보 은닉을 구현할 수 있다.
- ES6 모듈은 파일 단위의 모듈 스코프를 제공해 전역 오염을 줄인다.

4. 네임스페이스 객체 방식은 어떤 레거시 환경에서 여전히 유용할까?
5. 모듈 패턴의 private 변수는 클로저와 어떤 방식으로 연결될까?
6. 전역 변수 문제를 ESLint 규칙으로 막으려면 어떤 규칙이 필요할까?
7. 브라우저의 `window`, Node.js의 `global`, 공통의 `globalThis`는 언제 직접 사용할까?
8. ES6 모듈과 번들러(Vite/Webpack)의 모듈 처리는 어떤 차이가 있을까?

## 답변

1. **전역 변수를 “반드시” 써야 하는 경우**
   - 정말로 “전역”이어야 하는 경우는 드물고, 보통은 **전역처럼 보이는 단일 진입점/공유 채널**이 필요한 경우다.
   - 예: 레거시 환경에서 여러 스크립트가 느슨하게 결합되어 있고, 서로 통신할 **공용 훅(디버그 플래그, 진단용 핸들)**이 필요할 때.
   - 다만 가능하면 전역 변수 대신 **모듈 export**, **의존성 주입**, **전역 이벤트/메시지 버스** 같은 더 좁은 결합으로 대체하는 게 안전하다.

2. **전역 상태 vs 애플리케이션 상태 구분 기준**
   - **전역 상태(환경/플랫폼 레벨)**: 런타임 자체가 제공하는 값/서비스(예: `location`, 전역 이벤트, 환경 설정)처럼 “앱이 소유하지 않는 것”.
   - **애플리케이션 상태(도메인/제품 레벨)**: 사용자/화면/데이터 흐름 등 “앱이 소유하고 업데이트 규칙을 갖는 것”.
   - 실무적으로는 “도메인 상태는 상태 관리/스토어(또는 컴포넌트 state)로, 환경 접근은 어댑터(모듈)로 감싼다”처럼 **경계 레이어**를 두면 섞이지 않는다.

3. **ES6 모듈 시대에 IIFE가 여전히 유효한가**
   - 모듈을 쓸 수 있으면 대부분의 경우 IIFE의 역할(전역 오염 방지)은 **모듈 스코프**가 대체한다.
   - 그래도 가치가 남는 경우:
     - 빌드/번들 없이 바로 실행하는 **단일 HTML 스니펫**(데모/샌드박스)에서 지역 스코프가 필요할 때
     - 레거시 코드에서 파일 단위 모듈화를 당장 못 할 때, “최소 변경으로 전역 오염만 먼저 줄이기”에 유용

4. **네임스페이스 객체가 유용한 레거시 환경**
   - ES6 모듈/번들러 없이 여러 스크립트를 `<script>`로 로드하는 구조에서, 전역 식별자 충돌을 줄이기 위한 현실적 타협으로 쓰인다.
   - 예: 플러그인처럼 “외부가 특정 전역 이름(`MYAPP`)을 찾아야 하는” 통합 방식일 때.
   - 단, 네임스페이스 객체도 결국 전역이므로 “최소한의 표면적”만 남기고 내부는 캡슐화하는 식으로 사용한다.

5. **모듈 패턴의 private 변수와 클로저 연결**
   - 모듈 패턴은 IIFE 내부에 `num` 같은 변수를 두고, 바깥으로는 메서드만 반환한다.
   - 반환된 메서드들은 생성 당시의 렉시컬 환경(스코프)을 계속 참조하므로, IIFE 실행이 끝나도 `num`을 “기억”한다.
   - 이 “함수가 상위 스코프 변수를 붙잡고 살아있는” 현상이 바로 **클로저**이고, 그 결과 private 변수처럼 동작한다.

6. **전역 변수 문제를 줄이는 ESLint 규칙(대표 방향)**
   - 핵심은 “암묵적 전역/전역 오염/재할당”을 막는 것.
   - 보통 조합:
     - **정의되지 않은 변수 사용 금지**(암묵적 전역 생성 방지)
     - **전역 스코프에서 `var` 사용 제한**(중복 선언/전역 객체 프로퍼티화 위험 감소)
     - 필요 시 **특정 전역만 허용**(예: 테스트 환경의 `describe`, `it` 등)하고 나머지는 금지
   - 팀에서 “전역 사용은 예외적으로만” 허용하도록 정책화하는 게 규칙보다 더 효과적일 때가 많다.

7. **`window` / `global` / `globalThis`를 직접 쓰는 시점**
   - 런타임을 가로지르는(브라우저/Node/Worker) 코드를 작성할 때는 **`globalThis`**가 가장 안전한 공통 진입점이다.
   - **`window`**는 “브라우저 DOM 환경임이 확실하고, DOM API 접근이 필요”할 때만 직접 쓴다.
   - **`global`**은 Node 전용 코드에서 전역을 다뤄야 할 때 쓰되, 가능한 한 모듈 스코프/DI로 대체하는 편이 낫다.

8. **ES6 모듈 vs 번들러 모듈 처리 차이**
   - **ES6 모듈(브라우저 런타임 기능)**: 브라우저가 `import/export`를 직접 로딩/해석하고, 파일 단위 모듈 스코프를 제공한다.
   - **번들러(Vite/Webpack)**: 개발/배포 편의를 위해 모듈을 분석해 **번들링/코드 분할/트랜스파일** 등을 수행한다(호환성, 성능 최적화, 자산 처리 포함).
   - 실무 차이는 “브라우저가 그대로 이해할 수 없는 문법/리소스까지 포함해 프로젝트 전체를 패키징/최적화하느냐”에 있다.
