## 5. 표현식과 문

## 5.1 값

- 값(`value`)은 **식(표현식)이 평가되어 생성된 결과**다.
- 평가(`evaluate`)는 식을 해석해 값을 생성하거나 참조하는 과정이다.
- 예: `10 + 20`은 평가되어 `30`이라는 값을 만든다.
- 모든 값은 타입을 가지며 메모리에 비트 형태로 저장된다.
- 변수에 저장되는 것도 결국 값이다.

```js
var sum = 10 + 20; // sum에 저장되는 것은 식 자체가 아니라 평가 결과 30
```

## 5.2 리터럴

- 리터럴(`literal`)은 사람이 이해할 수 있는 문자/기호로 **값을 표현하는 표기법**이다.
- 자바스크립트 엔진은 런타임에 리터럴을 평가해 값을 생성한다.

```js
3; // 숫자 리터럴
("hello"); // 문자열 리터럴
true; // 불리언 리터럴
null; // null 리터럴
{
  a: 1;
} // 객체 리터럴
[1, 2, 3]; // 배열 리터럴
```

## 5.3 표현식

- 표현식(`expression`)은 **값으로 평가될 수 있는 문**이다.
- 즉 표현식은 새로운 값을 만들거나 기존 값을 참조한다.
- 리터럴 자체도 값으로 평가되므로 표현식이다.

```js
var score = 100; // 100은 표현식
var total = 50 + 50; // 50 + 50은 표현식
score; // 식별자 참조도 표현식
```

- 표현식은 리터럴, 식별자, 연산자, 함수 호출 등을 조합해 구성할 수 있다.
- 표현식과 그 평가 결과는 동치처럼 다룰 수 있다.
  - 예: `1 + 2`와 `3`

## 5.4 문

- 문(`statement`)은 프로그램을 구성하는 **기본 단위이자 최소 실행 단위**다.
- 프로그램은 문의 집합이며, 문은 토큰들의 조합으로 이루어진다.
- 문이 실행되면 어떤 동작이 발생한다.

대표적인 문:

- 선언문: `var x;`
- 할당문: `x = 5;`
- 조건문: `if (x > 1) { ... }`
- 반복문: `for (...) { ... }`
- 함수 선언문: `function foo() {}`

## 5.5 세미콜론과 ASI

- 세미콜론(`;`)은 문의 종료를 나타낸다.
- 자바스크립트는 ASI(세미콜론 자동 삽입) 기능이 있어 일부 경우 생략해도 동작한다.
- 하지만 ASI는 개발자 의도와 다르게 동작할 수 있어 예기치 않은 버그를 만들 수 있다.
- 실무/도구(예: ESLint 기본 규칙)에서는 보통 **세미콜론 명시**를 권장한다.

ASI 오동작 대표 예:

```js
function foo() {
  return;
  {
  }
}
console.log(foo()); // undefined (return; 로 해석)
```

## 5.6 표현식인 문과 표현식이 아닌 문

- 문은 두 가지로 나뉜다.
  - **표현식인 문**: 값으로 평가될 수 있는 문
  - **표현식이 아닌 문**: 값으로 평가될 수 없는 문

예:

```js
var x; // 표현식이 아닌 문(선언문)
x = 100; // 표현식인 문(할당문)
```

- 가장 쉬운 구분법: 변수에 할당해 보기
  - 표현식인 문: 할당 가능
  - 표현식이 아닌 문: 할당 시 에러

```js
var foo = (x = 100); // 가능 (x = 100이 100으로 평가됨)
// var bar = var x; // SyntaxError
```

### 완료 값(Completion Value)

- 개발자 도구에서 표현식이 아닌 문 실행 시 보이는 `undefined`는 “완료 값”이다.
- 이는 표현식의 평가 결과와 다르며, 일반 값처럼 참조/할당할 수 없다.

---

## 내용 정리

- 값은 표현식 평가 결과이며, 변수에는 값이 저장된다.
- 리터럴은 값을 만드는 표기법이고, 리터럴 자체도 표현식이다.
- 표현식은 값으로 평가될 수 있는 코드 조각이다.
- 문은 실행 단위이며 선언문/할당문/조건문/반복문 등이 있다.
- 세미콜론 생략은 가능하지만 ASI 함정 때문에 명시가 안전하다.
- “표현식인 문 vs 아닌 문”은 변수 할당 가능 여부로 빠르게 구분할 수 있다.

## 의문점 제기

1. “표현식은 값이다”를 초보자가 직관적으로 이해하도록 설명하는 가장 좋은 예시는 무엇일까?
2. ASI 때문에 실제로 자주 발생하는 버그 패턴은 어떤 것들이 있을까?
3. 팀 규칙에서 세미콜론 사용을 강제할 때 Prettier와 ESLint 역할 분리는 어떻게 하는 게 좋을까?
4. 표현식 중심 코드(함수형 스타일)가 가독성에 유리한 경우와 불리한 경우의 경계는?
5. “표현식인 문”을 과도하게 체이닝하면 디버깅이 어려워지는데, 적절한 제한 기준은?
6. 완료 값(`undefined`)과 실제 평가 값을 개발자 도구에서 구분해 읽는 습관은 어떻게 들일까?
7. 조건식/할당식을 한 문장에 섞어 쓰는 패턴은 언제 허용하고 언제 금지해야 할까?
8. 문과 표현식 구분이 가장 크게 도움이 되는 디버깅 상황은 어떤 경우일까?
