## 11. 원시 값과 객체의 비교

자바스크립트의 데이터 타입은 크게 **원시 타입**과 **객체 타입**으로 나뉜다. 둘은 다음 세 가지 측면에서 다르다.

| 구분 | 원시 값 | 객체 |
|------|---------|------|
| 변경 가능 여부 | 변경 불가능한 값 | 변경 가능한 값 |
| 변수에 저장되는 값 | 실제 값 | 참조 값 |
| 다른 변수에 할당 | 원시 값이 복사됨 | 참조 값이 복사됨 |

## 11.1 원시 값

### 11.1.1 변경 불가능한 값

원시 값은 변경 불가능한 값이다. 여기서 중요한 점은 **변수가 아니라 값이 변경 불가능하다**는 것이다.

```js
var score = 80;
score = 90;
```

위 코드는 숫자 값 `80` 자체를 `90`으로 바꾸는 것이 아니다. 새로운 숫자 값 `90`을 만들고, 변수 `score`가 새 값을 가리키도록 바꾸는 것이다.

즉, 변수는 재할당할 수 있지만 원시 값 자체는 변경할 수 없다.

```js
const o = {};

// const는 재할당이 금지된 변수일 뿐,
// 객체 자체의 변경을 막지는 않는다.
o.a = 1;

console.log(o); // { a: 1 }
```

- 상수: 재할당이 금지된 변수
- 불변성: 값 자체를 변경할 수 없는 성질

원시 값은 불변성을 가지므로 예기치 않은 변경으로부터 안전하고, 데이터의 신뢰성을 보장한다.

### 11.1.2 문자열과 불변성

문자열은 원시 값이며 변경 불가능하다.

```js
var str = "Hello";
str = "world";
```

위 코드는 `"Hello"` 문자열을 수정하는 것이 아니라, 새로운 문자열 `"world"`를 만들고 `str`이 그것을 가리키도록 바꾸는 것이다.

문자열은 배열처럼 인덱스로 각 문자에 접근할 수 있다.

```js
var str = "string";

console.log(str[0]);           // s
console.log(str.length);       // 6
console.log(str.toUpperCase()); // STRING
```

하지만 문자열은 원시 값이므로 일부 문자를 직접 변경할 수 없다.

```js
var str = "string";

str[0] = "S";

console.log(str); // string
```

문자열이 객체처럼 동작하는 이유는 원시 값을 객체처럼 사용할 때 일시적으로 래퍼 객체가 생성되기 때문이다.

### 11.1.3 값에 의한 전달

원시 값을 갖는 변수를 다른 변수에 할당하면 원시 값이 복사되어 전달된다.

```js
var score = 80;
var copy = score;

console.log(score, copy);      // 80 80
console.log(score === copy);   // true

score = 100;

console.log(score, copy);      // 100 80
console.log(score === copy);   // false
```

`copy = score`에서 `score`의 값 `80`이 `copy`에 복사된다. 이후 `score`를 재할당해도 `copy`에는 영향을 주지 않는다.

중요한 점:

- 원시 값은 서로 독립된 값처럼 동작한다.
- 한쪽 변수를 재할당해도 다른 변수에 영향을 주지 않는다.
- “값에 의한 전달”이라는 표현은 엄밀히 말하면 메모리 주소 전달 방식까지 포함하면 단순하지 않지만, 결과적으로 원시 값은 서로 간섭하지 않는다.

## 11.2 객체

객체는 프로퍼티 개수가 정해져 있지 않고, 동적으로 추가/삭제할 수 있다. 프로퍼티 값에도 제약이 없다.

따라서 객체는 원시 값처럼 필요한 메모리 크기를 미리 단순하게 정하기 어렵고, 관리 방식도 더 복잡하다.

### 11.2.1 변경 가능한 값

객체는 변경 가능한 값이다.

```js
var person = {
  name: "Lee"
};
```

객체를 변수에 할당하면 변수에는 객체 자체가 아니라 **참조 값**이 저장된다. 참조 값은 실제 객체가 저장된 메모리 공간을 가리키는 값이다.

```js
var person = {
  name: "Lee"
};

console.log(person); // { name: "Lee" }
```

객체는 재할당 없이 직접 변경할 수 있다.

```js
var person = {
  name: "Lee"
};

person.name = "Kim";
person.address = "Seoul";

console.log(person); // { name: "Kim", address: "Seoul" }
```

객체가 변경 가능한 값으로 설계된 이유:

- 객체는 크기가 일정하지 않다.
- 프로퍼티가 동적으로 추가/삭제될 수 있다.
- 객체를 매번 복사하면 메모리와 성능 비용이 크다.

하지만 이 구조에는 단점이 있다. 여러 식별자가 하나의 객체를 공유할 수 있고, 이로 인해 한쪽의 변경이 다른 쪽에 영향을 줄 수 있다.

### 얕은 복사와 깊은 복사

객체 안에 객체가 중첩되어 있을 때, 복사 방식은 크게 두 가지로 나뉜다.

- 얕은 복사: 한 단계만 복사하고 중첩 객체는 참조를 공유
- 깊은 복사: 중첩 객체까지 모두 복사

```js
const o = { x: { y: 1 } };

const c1 = { ...o };

console.log(c1 === o);     // false
console.log(c1.x === o.x); // true
```

위 예제에서 `c1`과 `o`는 다른 객체지만, 내부 객체 `x`는 같은 객체를 공유한다. 이것이 얕은 복사다.

깊은 복사는 중첩 객체까지 모두 새로 복사한다.

```js
const o = { x: { y: 1 } };
const c2 = structuredClone(o);

console.log(c2 === o);     // false
console.log(c2.x === o.x); // false
```

## 11.2.2 참조에 의한 전달

객체를 가리키는 변수를 다른 변수에 할당하면 참조 값이 복사된다.

```js
var person = {
  name: "Lee"
};

var copy = person;
```

이때 `person`과 `copy`는 같은 객체를 가리킨다.

```js
var person = {
  name: "Lee"
};

var copy = person;

console.log(copy === person); // true

copy.name = "Kim";
person.address = "Seoul";

console.log(person); // { name: "Kim", address: "Seoul" }
console.log(copy);   // { name: "Kim", address: "Seoul" }
```

객체를 공유하면 한쪽에서 프로퍼티를 변경했을 때 다른 쪽에서도 변경된 결과가 보인다.

정리하면:

- 원시 값 할당: 값이 복사되어 서로 독립적으로 동작
- 객체 할당: 참조 값이 복사되어 같은 객체를 공유

단, 자바스크립트에는 다른 언어의 포인터 기반 “참조에 의한 전달”과 완전히 같은 개념은 없다. 실제로는 변수에 저장된 값(원시 값 또는 참조 값)이 복사되어 전달된다고 이해하는 것이 더 정확하다.

## 객체 비교

객체를 비교할 때는 객체의 내용이 아니라 참조 값을 비교한다.

```js
var person1 = {
  name: "Lee"
};

var person2 = {
  name: "Lee"
};

console.log(person1 === person2);       // false
console.log(person1.name === person2.name); // true
```

`person1`과 `person2`는 내용은 같지만 서로 다른 객체다. 따라서 참조 값이 다르므로 `false`다.

반면 `person1.name`과 `person2.name`은 모두 원시 값 `"Lee"`로 평가되므로 `true`다.

---

## 내용 정리

- 원시 값은 변경 불가능하고, 객체는 변경 가능하다.
- 원시 값을 변수에 할당하면 실제 값이 저장된다.
- 객체를 변수에 할당하면 객체 자체가 아니라 참조 값이 저장된다.
- 원시 값을 다른 변수에 할당하면 값이 복사되어 서로 영향을 주지 않는다.
- 객체를 다른 변수에 할당하면 참조 값이 복사되어 같은 객체를 공유한다.
- 문자열은 원시 값이므로 일부 문자를 직접 변경할 수 없다.
- 얕은 복사는 중첩 객체의 참조를 공유하고, 깊은 복사는 중첩 객체까지 복사한다.
- 객체 비교는 내용 비교가 아니라 참조 값 비교다.

## 의문점 제기

1. “원시 값은 변경 불가능하지만 변수는 재할당 가능하다”를 헷갈리지 않게 설명하는 가장 좋은 예시는 무엇일까?
2. `const`로 선언한 객체의 프로퍼티 변경이 가능한 점은 초보자에게 어떤 오해를 만들 수 있을까?
3. 객체 공유로 인한 사이드 이펙트를 줄이려면 어떤 상황에서 복사를 선택해야 할까?
4. 얕은 복사와 깊은 복사의 비용 차이는 실제 서비스 코드에서 언제 문제가 될까?
5. `structuredClone`, 스프레드 문법, 라이브러리의 `cloneDeep`은 각각 어떤 기준으로 선택해야 할까?
6. 객체를 직접 변경하는 방식과 불변성을 유지하는 방식은 어떤 트레이드오프가 있을까?
7. 객체의 내용이 같은지 비교하려면 `===` 대신 어떤 방식이 필요할까?
8. 상태 관리 라이브러리에서 불변성을 강조하는 이유는 객체의 참조 공유 문제와 어떻게 연결될까?
