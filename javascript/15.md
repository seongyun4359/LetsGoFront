## 15. `let`, `const` 키워드와 블록 레벨 스코프

ES5까지 변수를 선언하는 유일한 방법은 `var` 키워드였다. 하지만 `var`는 중복 선언, 함수 레벨 스코프, 변수 호이스팅으로 인해 의도치 않은 문제를 만들 수 있다. ES6에서는 이를 보완하기 위해 `let`, `const`가 도입되었다.

## 15.1 `var` 키워드로 선언한 변수의 문제점

### 15.1.1 변수 중복 선언 허용

`var` 키워드로 선언한 변수는 같은 스코프 안에서 중복 선언할 수 있다.

```js
var x = 1;
var y = 1;

var x = 100;
var y;

console.log(x); // 100
console.log(y); // 1
```

초기화문이 있는 중복 선언은 기존 변수에 값을 다시 할당하는 것처럼 동작한다. 초기화문이 없는 중복 선언은 무시된다.

이 때문에 이미 선언된 변수를 모르고 다시 선언하면 의도치 않게 값이 바뀔 수 있다.

### 15.1.2 함수 레벨 스코프

`var`는 함수 몸체만 지역 스코프로 인정한다. `if`, `for` 같은 블록은 지역 스코프를 만들지 않는다.

```js
var x = 1;

if (true) {
  var x = 10;
}

console.log(x); // 10
```

`for` 문에서도 같은 문제가 생긴다.

```js
var i = 10;

for (var i = 0; i < 5; i++) {
  console.log(i); // 0 1 2 3 4
}

console.log(i); // 5
```

`var`의 함수 레벨 스코프는 전역 변수를 남발하거나 의도치 않은 중복 선언을 만들 가능성을 높인다.

### 15.1.3 변수 호이스팅

`var`로 선언한 변수는 변수 선언이 스코프의 선두로 끌어올려진 것처럼 동작한다. 선언문 이전에 참조할 수 있지만, 할당 전에는 `undefined`다.

```js
console.log(foo); // undefined

foo = 123;

console.log(foo); // 123

var foo;
```

변수 선언문 이전에 변수를 참조할 수 있다는 점은 코드 흐름을 읽기 어렵게 만들고 오류 가능성을 높인다.

## 15.2 `let` 키워드

`let`은 `var`의 단점을 보완하기 위해 도입된 변수 선언 키워드다.

### 15.2.1 변수 중복 선언 금지

`let`은 같은 스코프에서 중복 선언을 허용하지 않는다.

```js
let bar = 123;
// let bar = 456; // SyntaxError
```

### 15.2.2 블록 레벨 스코프

`let`으로 선언한 변수는 모든 코드 블록을 지역 스코프로 인정한다.

```js
let foo = 1;

{
  let foo = 2;
  let bar = 3;
}

console.log(foo); // 1
// console.log(bar); // ReferenceError
```

`if`, `for`, `while`, `try/catch` 등의 블록 안에서 선언한 `let` 변수는 해당 블록 내부에서만 유효하다.

```js
let i = 10;

function foo() {
  let i = 100;

  for (let i = 1; i < 3; i++) {
    console.log(i); // 1 2
  }

  console.log(i); // 100
}

foo();
console.log(i); // 10
```

### 15.2.3 변수 호이스팅과 TDZ

`let`으로 선언한 변수는 호이스팅이 발생하지 않는 것처럼 동작한다.

```js
// console.log(foo); // ReferenceError
let foo;
```

하지만 실제로는 `let`도 호이스팅된다. 다만 `var`와 달리 선언 단계와 초기화 단계가 분리되어 있다.

- `var`: 선언 단계와 초기화 단계가 런타임 이전에 함께 실행됨
- `let`: 선언 단계는 런타임 이전에 실행되지만, 초기화 단계는 변수 선언문에 도달했을 때 실행됨

스코프 시작 지점부터 초기화 단계 전까지 변수를 참조할 수 없는 구간을 **일시적 사각지대(TDZ, Temporal Dead Zone)** 라고 한다.

```js
// TDZ
// console.log(foo); // ReferenceError

let foo;
console.log(foo); // undefined

foo = 1;
console.log(foo); // 1
```

`let`도 호이스팅된다는 점은 다음 예제로 확인할 수 있다.

```js
let foo = 1;

{
  // 전역 foo를 참조하지 않고, 블록 내부의 let foo가 호이스팅되어 TDZ에 걸린다.
  // console.log(foo); // ReferenceError
  let foo = 2;
}
```

자바스크립트의 모든 선언(`var`, `let`, `const`, `function`, `class` 등)은 호이스팅된다. 단, `let`, `const`, `class`는 TDZ 때문에 호이스팅이 발생하지 않는 것처럼 보인다.

### 15.2.4 전역 객체와 `let`

브라우저 환경에서 `var`로 선언한 전역 변수는 전역 객체 `window`의 프로퍼티가 된다.

```js
var x = 1;

console.log(window.x); // 1
console.log(x);        // 1
```

하지만 `let`으로 선언한 전역 변수는 전역 객체의 프로퍼티가 아니다.

```js
let x = 1;

console.log(window.x); // undefined
console.log(x);        // 1
```

`let` 전역 변수는 전역 렉시컬 환경의 선언적 환경 레코드에 존재한다.

## 15.3 `const` 키워드

`const`는 상수를 선언하기 위해 사용한다. 대부분의 특징은 `let`과 동일하지만, 중요한 차이가 있다.

### 15.3.1 선언과 초기화

`const`는 반드시 선언과 동시에 초기화해야 한다.

```js
const foo = 1;
```

```js
// const bar; // SyntaxError
```

`const`도 블록 레벨 스코프를 가지며, TDZ로 인해 호이스팅이 발생하지 않는 것처럼 동작한다.

```js
{
  // console.log(foo); // ReferenceError
  const foo = 1;
  console.log(foo); // 1
}

// console.log(foo); // ReferenceError
```

### 15.3.2 재할당 금지

`const`로 선언한 변수는 재할당할 수 없다.

```js
const foo = 1;

// foo = 2; // TypeError
```

### 15.3.3 상수

`const`에 원시 값을 할당하면 재할당할 수 없고, 원시 값 자체도 변경 불가능하므로 상수처럼 사용할 수 있다.

매직 넘버 대신 의미 있는 상수를 사용하면 가독성과 유지보수성이 좋아진다.

```js
const TAX_RATE = 0.1;

let preTaxPrice = 100;
let afterTaxPrice = preTaxPrice + (preTaxPrice * TAX_RATE);

console.log(afterTaxPrice); // 110
```

상수 이름은 보통 대문자와 언더스코어를 사용한다.

### 15.3.4 `const` 키워드와 객체

`const`는 재할당을 금지할 뿐, 불변을 의미하지 않는다.

`const`로 선언한 변수에 객체를 할당하면 객체 자체는 변경할 수 있다.

```js
const person = {
  name: "Lee"
};

person.name = "Kim";

console.log(person); // { name: "Kim" }
```

객체의 프로퍼티를 변경해도 변수에 저장된 참조 값은 바뀌지 않으므로 재할당이 아니다.

## 15.4 `var` vs `let` vs `const`

권장 사용 방식:

1. ES6 이후에는 `var`를 사용하지 않는다.
2. 재할당이 필요한 경우에만 `let`을 사용한다.
3. 재할당이 필요 없는 값은 `const`를 사용한다.
4. 일단 `const`로 선언하고, 정말 재할당이 필요할 때 `let`으로 바꾼다.

```js
const user = {
  name: "Lee"
};

let count = 0;

count++;
```

`const`를 기본으로 사용하면 의도치 않은 재할당을 막을 수 있어 더 안전하다.

## 내용 정리

- `var`는 중복 선언을 허용하고 함수 레벨 스코프를 따르며, 호이스팅으로 `undefined` 참조가 가능하다.
- `let`은 중복 선언을 금지하고 블록 레벨 스코프를 따른다.
- `let`도 호이스팅되지만 TDZ 때문에 선언 전 참조가 불가능하다.
- `var` 전역 변수는 브라우저에서 `window`의 프로퍼티가 되지만, `let` 전역 변수는 그렇지 않다.
- `const`는 선언과 동시에 초기화해야 하며 재할당이 금지된다.
- `const`는 불변을 의미하지 않으므로 객체의 프로퍼티 변경은 가능하다.
- 상수는 의미 있는 이름으로 선언해 매직 넘버를 줄이고 유지보수성을 높인다.
- 기본은 `const`, 재할당이 필요할 때만 `let`, `var`는 지양한다.

## 의문점 제기

1. `let`과 `const`도 호이스팅되는데 “호이스팅되지 않는다”라고 설명하면 어떤 오해가 생길까?
2. TDZ는 실제 코드에서 어떤 버그를 미리 막아주는 역할을 할까?
3. `const` 객체의 프로퍼티 변경을 막으려면 어떤 방법(`Object.freeze`, 불변 라이브러리 등)이 필요할까?
4. 반복문에서 `let i`를 사용하면 클로저 동작에 어떤 차이가 생길까?
5. 전역 `let`이 `window` 프로퍼티가 아닌 점은 브라우저 디버깅에서 어떤 차이를 만들까?
6. 팀 컨벤션으로 `prefer-const`를 강제하면 실제 코드 품질에 어떤 장점이 있을까?
7. 재할당이 필요한 변수와 객체 내부 변경이 필요한 변수를 어떻게 구분해 네이밍하면 좋을까?
8. `var`를 사용하는 레거시 코드를 `let`/`const`로 바꿀 때 가장 조심해야 할 부분은 무엇일까?
