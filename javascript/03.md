## 3. 자바스크립트 개발 환경과 실행 방법

## 3.1 자바스크립트 실행 환경

- 브라우저와 Node.js 모두 자바스크립트 엔진을 내장하고 있어 자바스크립트를 실행할 수 있다.
- 다만 **목적이 다르다**:
  - 브라우저: HTML/CSS/JS를 실행해 화면 렌더링
  - Node.js: 브라우저 밖에서 JS 실행 환경 제공(주로 서버)
- 두 환경은 ECMAScript 코어는 공유하지만, **추가 API는 다르다**.
  - 브라우저: DOM, BOM, `fetch`, `requestAnimationFrame`, Web Storage 등 Web API 제공
  - Node.js: 파일 시스템(`fs`), `http` 등 Node 고유 API 제공
- 브라우저는 보안상 로컬 파일 시스템에 직접 접근할 수 없다(사용자가 선택한 파일 읽기 등 제한적 기능만 허용).
- Node.js는 DOM API를 기본 제공하지 않으므로, 서버 측 HTML 파싱/가공 시 `cheerio` 같은 라이브러리를 사용하기도 한다.

## 3.2 웹 브라우저(크롬) 활용

- 실습 환경으로 크롬을 주로 사용(ECMAScript 대응이 빠르고 점유율이 높음).
- **DevTools(개발자 도구)**는 웹 개발 필수 도구이며 별도 설치 없이 사용 가능.
- 주요 패널:
  - `Elements`: DOM/CSS 확인·임시 수정
  - `Console`: 에러 확인, `console.log` 출력, REPL 실행
  - `Sources`: 브레이크포인트·스텝 실행 등 디버깅
  - `Network`: 요청/응답 및 성능 관찰 -> API 관련 작업 시, 진짜 자주 봄.
  - `Application`: 스토리지/쿠키 확인 -> 중요!

### 콘솔과 REPL

- 콘솔은 에러를 가장 먼저 확인하는 장소다.
- `console.log(...)`로 중간 값을 확인하며 디버깅할 수 있다.
- 콘솔에서 JS를 직접 실행하는 REPL처럼 사용할 수 있다.

### 브라우저 디버깅 핵심 흐름

1. 콘솔 에러 메시지 확인
2. 에러 위치(Sources)로 이동
3. 브레이크포인트 설정 후 재현
4. 변수 값(예: `null`) 확인
5. 원인 수정 후 재실행

예시로 `document.getElementById('counter-x')`처럼 잘못된 선택자를 쓰면 `null`이 되어, 이후 `innerHTML` 접근에서 오류가 발생한다.

## 3.3 Node.js

- 프로젝트가 커지면 프레임워크(React/Angular 등)와 도구(Babel/Webpack/ESLint 등)를 쓰게 되고, 이때 Node.js와 npm이 필요하다.
- **Node.js**: V8 기반 자바스크립트 런타임.
- **npm**: 패키지 저장소 + 설치/관리 CLI.

### 설치와 확인

- 일반적으로 LTS(안정)와 Current(최신) 중 선택 가능.
- 설치 후 버전 확인:
  - `node -v`
  - `npm -v`

### Node.js REPL

- `node` 명령으로 REPL 진입 후 간단한 JS 코드를 즉시 실행 가능.
- 파일 실행은 `node index.js` 형태로 사용(`.js` 생략 가능).

## 3.4 VS Code에서 실행하기

- 콘솔/REPL만으로는 한계가 있으므로, 실무에서는 코드 에디터를 사용한다.
- VS Code는 자동완성, 에러 표시, 디버깅, Git 연동 등 개발 생산성을 높여준다.

### 내장 터미널

- VS Code 내장 터미널에서 `node index`로 파일 실행 가능.

### Code Runner 확장

- 단축키로 현재 파일을 빠르게 실행할 수 있다.
- 단, 실행 환경이 Node.js이므로 브라우저 전용 API(`alert`, `document` 등)는 오류가 난다.

### Live Server 확장

- 브라우저 전용 API가 포함된 코드는 브라우저에서 실행해야 한다.
- `index.html`로 진입점을 만들고 Live Server로 실행하면 저장 시 자동 새로고침되어 학습/개발이 편하다.

## 내용 정리

- 자바스크립트는 브라우저와 Node.js에서 실행되지만, **API 호환 범위가 다르다**.
- 브라우저는 **화면/DOM/Web API**, Node.js는 **서버/파일 시스템/런타임 API** 중심이다.
- 디버깅은 콘솔 에러 확인 → Sources 브레이크포인트 → 값 추적의 순서가 기본이다.
- Node.js와 npm은 현대 프런트엔드 도구 체인과 생태계의 기반이다.
- VS Code에서는 Node 실행(Code Runner)과 브라우저 실행(Live Server)을 목적에 맞게 구분해야 한다.

## 의문점 제기

1. 브라우저 API와 Node API를 실무에서 가장 빠르게 구분하는 체크리스트는 무엇일까?
2. 학습 단계에서 LTS와 Current 중 어떤 버전을 기준으로 삼는 것이 더 효율적일까?
3. Code Runner로 빠르게 확인할 코드와 브라우저에서 반드시 확인할 코드는 어떻게 나눌까?
4. 디버깅 시 `console.log` 중심 접근과 브레이크포인트 중심 접근은 어떤 기준으로 선택할까?
5. 프런트엔드 프로젝트에서 npm 의존성은 어느 수준으로 잠그는 것이(버전 고정) 안전할까?
6. Live Server와 실제 배포 서버 환경 차이를 줄이려면 어떤 추가 설정이 필요할까?
7. SSR/CSR 개념을 학습할 때, 현재 단계에서 최소한으로 알아야 할 경계는 어디까지일까?
8. 브라우저 보안 모델(CORS, 파일 접근 제한 등)을 초보가 먼저 이해해야 할 핵심 포인트는 무엇일까?
