## 16. 프로퍼티 어트리뷰트

## 16.1 내부 슬롯과 내부 메서드

- 내부 슬롯(Internal Slot)과 내부 메서드(Internal Method)는 ECMAScript 사양에서 엔진 동작을 설명하기 위한 개념이다.
- 표기: `[[...]]` 형태(예: `[[Prototype]]`, `[[Get]]`).
- 자바스크립트 코드에서 직접 접근/호출할 수는 없다.
- 다만 일부는 간접 접근 가능하다. 예: `[[Prototype]]` <-> `__proto__`.

```js
const o = {};

// 직접 접근 불가
// o.[[Prototype]] // SyntaxError

// 간접 접근 가능
console.log(o.__proto__ === Object.prototype); // true
```

## 16.2 프로퍼티 어트리뷰트와 프로퍼티 디스크립터 객체

- 프로퍼티가 생성될 때 엔진은 내부적으로 어트리뷰트를 기본값으로 정의한다.
- 데이터 프로퍼티 기준 핵심 어트리뷰트
  - `[[Value]]`
  - `[[Writable]]`
  - `[[Enumerable]]`
  - `[[Configurable]]`
- 확인 방법: `Object.getOwnPropertyDescriptor(obj, key)`
- 전체 확인: `Object.getOwnPropertyDescriptors(obj)`

```js
const person = { name: "Lee" };
person.age = 20;

console.log(Object.getOwnPropertyDescriptor(person, "name"));
console.log(Object.getOwnPropertyDescriptors(person));
```

## 16.3 데이터 프로퍼티와 접근자 프로퍼티

### 16.3.1 데이터 프로퍼티

- 일반적인 key-value 프로퍼티.
- 생성 시 기본값:
  - `value`: 실제 값
  - `writable`: `true`
  - `enumerable`: `true`
  - `configurable`: `true`

### 16.3.2 접근자 프로퍼티

- 자체 값을 직접 저장하지 않고, getter/setter 함수로 동작한다.
- 어트리뷰트:
  - `[[Get]]`
  - `[[Set]]`
  - `[[Enumerable]]`
  - `[[Configurable]]`

```js
const person = {
  firstName: "Ungmo",
  lastName: "Lee",

  get fullName() {
    return `${this.firstName} ${this.lastName}`;
  },

  set fullName(name) {
    [this.firstName, this.lastName] = name.split(" ");
  },
};

person.fullName = "Heegun Lee";
console.log(person.fullName); // Heegun Lee
```

## 16.4 프로퍼티 정의

- `Object.defineProperty()`로 프로퍼티를 명시적으로 정의/재정의할 수 있다.
- 디스크립터에서 생략 시 기본값:
  - `value/get/set`: `undefined`
  - `writable/enumerable/configurable`: `false`
- 여러 개를 한 번에 정의: `Object.defineProperties()`

```js
const person = {};

Object.defineProperty(person, "firstName", {
  value: "Ungmo",
  writable: true,
  enumerable: true,
  configurable: true,
});

Object.defineProperty(person, "lastName", {
  value: "Lee", // 나머지 생략 -> writable/enumerable/configurable 모두 false
});
```

## 16.5 객체 변경 방지

### 비교

- `Object.preventExtensions(obj)`
  - 프로퍼티 추가 금지
- `Object.seal(obj)`
  - 추가/삭제 금지 + 재정의 금지
- `Object.freeze(obj)`
  - 추가/삭제/재정의/값 변경 금지(읽기 전용)

### 16.5.1 객체 확장 금지

```js
const person = { name: "Lee" };
Object.preventExtensions(person);

person.age = 20; // 무시(엄격 모드면 에러)
delete person.name; // 삭제는 가능
```

### 16.5.2 객체 밀봉

```js
const person = { name: "Lee" };
Object.seal(person);

person.name = "Kim"; // 값 변경 가능
delete person.name; // 무시(엄격 모드면 에러)
```

### 16.5.3 객체 동결

```js
const person = { name: "Lee" };
Object.freeze(person);

person.name = "Kim"; // 무시(엄격 모드면 에러)
```

### 16.5.4 불변 객체(깊은 동결)

- `Object.freeze()`는 얕은 동결만 수행한다.
- 중첩 객체까지 막으려면 재귀적으로 동결해야 한다.

```js
function deepFreeze(target) {
  if (target && typeof target === "object" && !Object.isFrozen(target)) {
    Object.freeze(target);
    Object.keys(target).forEach((key) => deepFreeze(target[key]));
  }
  return target;
}
```

## 핵심 정리

- 프로퍼티 동작은 내부 어트리뷰트로 결정된다.
- 데이터 프로퍼티와 접근자 프로퍼티는 디스크립터 구조가 다르다.
- `defineProperty` 계열로 어트리뷰트를 정밀 제어할 수 있다.
- 객체 변경 방지는 목적에 따라 `preventExtensions` → `seal` → `freeze` 순으로 강해진다.
- 완전한 불변 객체가 필요하면 `deepFreeze`가 필요하다.

## 의문점 제기(생각해볼 질문)

- `[[Prototype]]` 같은 내부 슬롯은 왜 직접 접근을 막고 간접 접근만 허용할까?
- `Object.defineProperty()`의 기본값이 왜 `false` 중심으로 설계되었을까?
- 접근자 프로퍼티(get/set)는 편리하지만, 과도하게 사용하면 디버깅 난이도가 올라가지 않을까?
- `enumerable: false` 프로퍼티는 어떤 상황(보안/캡슐화/라이브러리 설계)에서 유용할까?
- `freeze`가 얕은 동결만 제공하는 이유는 성능 때문일까, 명세 단순성 때문일까?
- `deepFreeze`는 순환 참조 객체에서 어떻게 동작해야 안전할까?
- 불변성을 지키려면 `deepFreeze`와 `Immer` 같은 라이브러리 중 어떤 전략이 더 실용적일까?
- 실무에서 `seal`은 생각보다 잘 안 쓰이는데, 어떤 도메인에서 가장 효과적일까?

## 의문점 답변

- 내부 슬롯 직접 접근을 막는 이유는 엔진 최적화와 명세 안정성 보장을 위해서다. 외부 코드가 내부 구조를 임의로 건드리면 언어 일관성이 깨진다.
- `defineProperty` 기본값이 `false`인 이유는 "명시하지 않은 권한은 열지 않는다"는 보수적 설계 때문이다.
- get/set 남용은 추적을 어렵게 만든다. 단순 값 필드는 데이터 프로퍼티, 검증/파생값 계산이 필요할 때만 접근자 프로퍼티를 쓰는 것이 좋다.
- `enumerable: false`는 내부 상태 은닉, 로깅 노이즈 감소, 라이브러리 보조 메서드 숨김에 유용하다.
- `freeze`가 얕은 동결인 이유는 성능 비용과 순환 참조 처리 복잡도를 언어 기본 동작에서 강제하지 않기 위해서다.
- 순환 참조를 안전하게 처리하려면 `WeakSet`으로 방문 객체를 기록하며 재귀 동결해야 한다.
- 작은 설정 객체는 `deepFreeze`, 복잡한 상태 업데이트가 많은 앱 상태는 `Immer`가 실용적이다.
- `seal`은 플러그인 설정 객체, API 응답 스키마 고정, 런타임에 키 추가를 막아야 하는 도메인에서 효과적이다.
