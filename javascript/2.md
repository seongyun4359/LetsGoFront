# 2. 자바스크립트란?

## 2.1 자바스크립트의 탄생

- 1995년, 넷스케이프 커뮤니케이션즈는 웹페이지 보조 기능을 위해 브라우저에서 동작하는 경량 프로그래밍 언어 도입을 결정했고, 브렌던 아이크(Brendan Eich)가 자바스크립트를 개발했다.
- 1996년 3월 넷스케이프 내비게이터 2.0에 탑재. 이름은 Mocha → (같 해 9월) LiveScript → (12월) JavaScript로 최종 확정.
- 이후 자바스크립트는 모든 주요 브라우저의 표준처럼 자리 잡았다.  
  동시에 마이크로소프트의 JScript(파생) 등으로 분열·호환 이슈가 생기기도 했다.

## 2.2 자바스크립트의 표준화

- 1996년 8월, IE에 JScript 탑재. JScript와 자바스크립트는 표준이 아닌 상태에서 “적당히” 호환되며, 업체는 시장 점유를 위해 자사 브라우저 전용 기능을 경쟁적으로 추가 → 크로스 브라우징 문제 심화.
- 1996년 11월, 넷스케이프는 ECMA 인터내셔널에 자바스크립트 표준화를 요청.
- 1997년 7월, ECMA-262 초판( ECMAScript 1 ) 사양 완성. 상표 이슈로 사양명은 ECMAScript로 정리(일상적으로는 “자바스크립트”라고도 부른다).
- 1999년 ES3, 2009년 ES5(HTML5와 함께 부각) 등으로 정리되며, 2015년 ES6(ECMAScript 2015)에서 `let`/`const`, 화살표 함수, 클래스, 모듈 등 대규모 문법·기능이 도입됨. ES6 이후는 작은 기능이 매년 추가되는 흐름(연도 기반 표기).

### ECMAScript 버전별 요약(교재 표 기준)

| 버전        | 출시 연도 | 특징(요지)                                                                     |
| ----------- | --------- | ------------------------------------------------------------------------------ |
| ES1         | 1997      | 초판                                                                           |
| ES2         | 1998      | ISO/IEC 16262와 맞춤                                                           |
| ES3         | 1999      | 정규식, `try`…`catch`                                                          |
| ES5         | 2009      | JSON, strict mode, 접근자 프로퍼티, 배열 메서드(`forEach`, `map`, `filter` 등) |
| ES6(ES2015) | 2015      | `let`/`const`, 클래스, 화살표, 모듈, `Promise`, `Map`/`Set`, `Proxy` 등        |
| ES2016      | 2016      | 지수(``) 연산, `Array.prototype.includes` 등                                   |
| ES2017      | 2017      | `async`/`await`, `Object` 정적 메서드 등                                       |
| ES2018      | 2018      | 객체 rest/spread, `Promise.finally`, `for await…of` 등                         |
| ES2019      | 2019      | `Object.fromEntries`, `flat`/`flatMap` 등                                      |
| ES2020      | 2020      | `matchAll`, `BigInt`, `globalThis`, 옵셔널 체이닝·null 병합 등                 |

## 2.3 자바스크립트 성장의 역사

- 초기: 웹 보조용. 대부분 로직은 서버, 브라우저는 HTML/CSS 렌더링 수준.
- 렌더링: HTML·CSS·JS를 해석해 화면에 그리는 과정(서버가 HTML로 만드는 SSR도 이 맥락에서 쓰인다).
- Ajax(1999): `XMLHttpRequest` 등으로 비동기로 서버와 데이터만 주고받고, 전체 HTML 재로딩 대신 필요한 부분만 갱신 → 반응성·UX 개선(깜빡임·불필요 통신 감소).
- 구글 맵스(2005): 브라우저에서 JS + Ajax로 데스크톱에 가까운 경험을 보여, “웹앱”으로서의 JS 가능성을 부각.
- jQuery(2006): DOM 조작·크로스 브라우징을 쉽게(당시엔 jQuery를 먼저 배우는 흐름도 있었음).
- V8(2008): 매우 빠른 JS 엔진 → 웹앱·풍부한 UX에 JS가 본격적으로 쓰이게 됨. 서버 쪽 로직이 클라이언트(브라우저)로 이전되며 프런트엔드가 주목.
- Node.js(2009): V8 기반 JS 런타임 — 브라우저 밖(주로 서버)에서도 JS 실행. 비동기 I/O, 이벤트 루프·단일 스레드 모델로 I/O 잦은 작업(SPA 등)에 잘 맞고, CPU 집약 작업에는 덜 권하는 편(교재 서술).
- 다양한 영역: Electron(데스크톱), React Native/하이브리드, TensorFlow.js 등 크로스 플랫폼·인기 언어로서 위상 상승.
- SPA 프레임워크: Angular, React, Vue, Svelte 등 컴포넌트 기반으로 대규모·복잡한 프런트엔드 구조를 짜기 쉬워짐.

## 2.4 자바스크립트와 ECMAScript

- ECMAScript: ECMA-262 표준이라는 뜻으로, 값, 타입, 문법, 빌트인 객체 등 코어를 규정.
- 구현: 브라우저(및 Node 등)는 표준에 맞춰 자바스크립트 엔진을 탑재.
- “자바스크립트”라고 할 때 보통: ECMAScript 코어 + (브라우저에서) 클라이언트 Web API — DOM, BOM, `fetch`, Canvas, Web Storage, Worker 등( W3C 등 별도 사양). MDN Web API 문서로 확인.

## 2.5 자바스크립트의 특징

- HTML·CSS와 함께 웹 3요소 중, 브라우저에서 동작하는 주된 프로그래밍 언어.
- 문법엔 C·자바, 프로토타입·일급 함수 등은 셀프·스킴 등의 영향.
- “명시적 컴파일 없이” 인터프리터에 가깝게 쓰인다는 설명이 흔하지만, 최신 엔진(V8 등) 은 JIT 등으로 인터프리트 + (핫 패스) 컴파일·최적화를 섞는다. 전통적 컴파일/인터프리트 경계는 점차 모호(실행 파일을 따로 뽑는 것이 아닌 점 등으로 “순수 컴파일 언어”로 부르기 어렵다는 논의).
- 멀티 패러다임: 명령형, 함수형, 프로토타입 기반 객체지향(ES6 클래스는 이 위의 문법적 편의). “클래스 키워드가 약하다=객체지향이 약하다”는 오해에 주의.

## 2.6 ES6 브라우저 지원 현황(개념)

- 모던 브라우저는 ES6 대부분 지원하나 100%는 아님. IE·구형은 ES6 미지원이 많음.
- 최신 문법을 쓰되 구형까지 맞출 때는 Babel 등 트랜스파일로 ES5 이하로 맞추는 방식(빌드 체인·폴리필과 함께).

## 내용 정리

- 자바스크립트는 1995년 넷스케이프에서 웹 보조 스크립트로 시작해, Mocha → LiveScript → JavaScript로 이름이 바뀌었다.
- JScript와 브라우저 전쟁으로 표준이 흐트러져 ECMAScript로 통일이 진행됐다.
- Ajax로 부분 갱신, V8으로 성능, Node.js로 서버·범용까지 확장, 프레임워크로 대규모 UI를 다루게 됐다.
- ECMAScript는 문법·코어, 브라우저 Web API는 별도 — 합쳐서 “실무의 자바스크립트”에 가깝다.
- 엔진은 JIT 등으로 빠른 실행에 가깝게 발전했고, 지원이 안 되는 환경은 Babel 등으로 다운그레이드한다.

## 의문점 제기

1. ECMAScript 연도 표기(ES2015 vs ES6)를 팀/문서에서 어떻게 통일하는 게 혼란이 적을까?
2. “인터프리터 언어”라는 설명과 V8 JIT 사이를 어떻게 말로 정리하면 초보자에게 가장 정확할까?
3. Web API는 ECMA-262에 없다 — Node와 브라우저의 전역 API 차이(`window` vs `global` 등)는 어디까지 표준/구현인지 구분하려면?
4. 크로스 브라우징이 예전과 달리 나아진 지금, Babel·Polyfill을 언제까지·어떤 퍼센트까지 써야 할까?
5. 프로토타입 기반 OOP와 class 문법 — 설계/리뷰에서 어떤 기준으로 하나로 맞출까?
6. CPU-bound vs I/O-bound를 Node에서 나눌 때, 실무에서 Worker·다른 언어 서비스는 어떤 기준으로 끼우는가?
7. SPA 프레임워크 선택(React, Vue, Svelte 등)은 “성능”인가 “팀/생태계”인가, 우선순위는?
8. 최신 ES 기능을 쓰면서 ESLint·TypeScript로 “지원/의미”를 잡는 워크플로는 어떻게 잡는 게 좋을까?
