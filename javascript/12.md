## 12. 함수

함수는 자바스크립트의 핵심 개념이다. 스코프, 실행 컨텍스트, 클로저, 생성자 함수, 메서드, `this`, 프로토타입, 모듈화 등 많은 개념이 함수와 연결된다.

## 12.1 함수란?

함수는 일련의 과정을 문으로 구현하고 코드 블록으로 감싸 하나의 실행 단위로 정의한 것이다.

```js
function add(x, y) {
  return x + y;
}

add(2, 5); // 7
```

함수의 구성 요소:

- 함수 이름: 함수를 식별하는 이름
- 매개변수: 함수 내부로 입력을 전달받는 변수
- 인수: 함수를 호출할 때 전달하는 실제 값
- 함수 몸체: 실행할 문들의 코드 블록
- 반환값: 함수 실행 결과로 외부에 반환되는 값

함수는 정의만으로 실행되지 않는다. 함수를 실행하려면 인수를 전달하며 명시적으로 호출해야 한다.

```js
var result = add(2, 5);
console.log(result); // 7
```

## 12.2 함수를 사용하는 이유

함수는 같은 작업을 여러 번 재사용할 수 있게 해준다.

```js
function add(x, y) {
  return x + y;
}

add(1, 2); // 3
add(3, 4); // 7
add(5, 6); // 11
```

함수를 사용하면:

- 코드 중복을 줄일 수 있다.
- 유지보수가 쉬워진다.
- 실수 가능성이 줄어든다.
- 적절한 함수 이름을 통해 코드의 의도를 드러낼 수 있다.

코드는 컴퓨터뿐 아니라 개발자를 위한 문서이기도 하므로, 함수 이름은 역할이 잘 드러나게 짓는 것이 중요하다.

## 12.3 함수 리터럴

자바스크립트의 함수는 객체 타입의 값이다. 따라서 함수도 리터럴로 생성할 수 있다.

```js
var f = function add(x, y) {
  return x + y;
};
```

함수 리터럴 구성 요소:

- `function` 키워드
- 함수 이름(생략 가능)
- 매개변수 목록
- 함수 몸체

함수 이름이 있는 함수는 기명 함수, 이름이 없는 함수는 익명 함수라고 한다.

함수는 객체지만 일반 객체와 다르다. 일반 객체는 호출할 수 없지만 함수는 호출할 수 있다.

## 12.4 함수 정의

함수 정의란 함수를 호출하기 전에 매개변수, 실행할 문, 반환할 값을 지정하는 것이다.

대표적인 함수 정의 방식:

```js
// 함수 선언문
function add(x, y) {
  return x + y;
}

// 함수 표현식
var add = function (x, y) {
  return x + y;
};

// Function 생성자 함수
var add = new Function("x", "y", "return x + y");

// 화살표 함수
const add = (x, y) => x + y;
```

### 12.4.1 함수 선언문

함수 선언문은 함수 이름을 생략할 수 없다.

```js
function add(x, y) {
  return x + y;
}

console.log(add(2, 5)); // 7
```

함수 선언문은 표현식이 아닌 문이다. 하지만 자바스크립트 엔진은 함수 선언문을 평가할 때 함수 이름과 같은 식별자를 암묵적으로 생성하고, 그 식별자에 함수 객체를 할당한다.

따라서 함수는 함수 이름으로 호출되는 것처럼 보이지만, 실제로는 함수 객체를 가리키는 식별자로 호출된다.

```js
function add(x, y) {
  return x + y;
}

// add는 함수 이름이자, 엔진이 암묵적으로 만든 식별자처럼 동작한다.
console.log(add(2, 5)); // 7
```

기명 함수 리터럴은 문맥에 따라 함수 선언문 또는 함수 리터럴 표현식으로 해석될 수 있다.

```js
function foo() {
  console.log("foo");
}

foo(); // foo

(function bar() {
  console.log("bar");
});

// bar(); // ReferenceError
```

`bar`는 함수 몸체 내부에서만 참조 가능한 함수 이름이므로 외부에서 호출할 수 없다.

### 12.4.2 함수 표현식

자바스크립트 함수는 일급 객체다. 즉, 값처럼 변수에 할당하거나 인수로 전달하거나 반환할 수 있다.

```js
var add = function (x, y) {
  return x + y;
};

console.log(add(2, 5)); // 7
```

함수 표현식에서는 함수 이름을 생략하는 익명 함수가 일반적이다.

기명 함수 표현식을 사용하더라도 함수 이름은 함수 몸체 내부에서만 유효하다.

```js
var add = function foo(x, y) {
  return x + y;
};

console.log(add(2, 5)); // 7
// console.log(foo(2, 5)); // ReferenceError
```

### 12.4.3 함수 생성 시점과 함수 호이스팅

함수 선언문은 런타임 이전에 함수 객체가 생성되고 식별자에 할당된다. 따라서 선언문 이전에도 호출할 수 있다.

```js
console.log(add(2, 5)); // 7

function add(x, y) {
  return x + y;
}
```

함수 표현식은 변수에 함수 리터럴을 할당하는 방식이다. 변수 선언은 호이스팅되지만 함수 객체 할당은 런타임에 이루어진다.

```js
// console.log(sub(2, 5)); // TypeError: sub is not a function

var sub = function (x, y) {
  return x - y;
};
```

정리:

- 함수 선언문: 함수 호이스팅 발생
- 함수 표현식: 변수 호이스팅만 발생

함수 호이스팅은 호출 전에 선언해야 한다는 흐름을 흐릴 수 있으므로, 함수 표현식을 선호하는 스타일도 있다.

### 12.4.4 Function 생성자 함수

`Function` 생성자 함수로도 함수를 만들 수 있다.

```js
var add = new Function("x", "y", "return x + y");

console.log(add(2, 5)); // 7
```

하지만 일반적이지 않고 권장되지 않는다. 함수 선언문/표현식과 다르게 클로저를 만들지 않는 등 동작 차이가 있다.

```js
var add1 = (function () {
  var a = 10;
  return function (x, y) {
    return x + y + a;
  };
})();

console.log(add1(1, 2)); // 13

var add2 = (function () {
  var a = 10;
  return new Function("x", "y", "return x + y + a;");
})();

// console.log(add2(1, 2)); // ReferenceError
```

### 12.4.5 화살표 함수

ES6에서 도입된 화살표 함수는 `function` 키워드 대신 `=>`를 사용한다.

```js
const add = (x, y) => x + y;

console.log(add(2, 5)); // 7
```

화살표 함수는 항상 익명 함수로 정의한다. 기존 함수를 완전히 대체하기 위한 문법은 아니며, 내부 동작도 다르다.

화살표 함수의 주요 특징:

- 생성자 함수로 사용할 수 없다.
- `this` 바인딩 방식이 일반 함수와 다르다.
- `prototype` 프로퍼티가 없다.
- `arguments` 객체를 생성하지 않는다.

## 12.5 함수 호출

함수는 함수 객체를 가리키는 식별자와 함수 호출 연산자 `()`로 호출한다.

```js
add(2, 5);
```

함수가 호출되면 실행 흐름이 함수 내부로 이동하고, 인수가 매개변수에 순서대로 할당된 뒤 함수 몸체가 실행된다.

### 12.5.1 매개변수와 인수

매개변수는 함수 정의 시 선언하고, 인수는 함수 호출 시 전달한다.

```js
function add(x, y) {
  return x + y;
}

var result = add(1, 2);
```

매개변수는 함수 몸체 내부에서만 참조할 수 있다.

```js
function add(x, y) {
  console.log(x, y);
  return x + y;
}

add(2, 5);
// console.log(x, y); // ReferenceError
```

자바스크립트 함수는 매개변수 개수와 인수 개수가 일치하는지 검사하지 않는다.

```js
function add(x, y) {
  return x + y;
}

console.log(add(2));        // NaN
console.log(add(2, 5, 10)); // 7
```

초과된 인수는 버려지는 것처럼 보이지만, 실제로는 `arguments` 객체에 보관된다.

```js
function add(x, y) {
  console.log(arguments);
  return x + y;
}

add(2, 5, 10);
```

### 12.5.2 인수 확인

자바스크립트는 동적 타입 언어이므로 함수 호출 시 인수 타입이나 개수를 자동으로 검사하지 않는다.

```js
function add(x, y) {
  return x + y;
}

console.log(add(2));        // NaN
console.log(add("a", "b")); // ab
```

필요하다면 함수 내부에서 직접 인수를 확인해야 한다.

```js
function add(x, y) {
  if (typeof x !== "number" || typeof y !== "number") {
    throw new TypeError("인수는 모두 숫자 값이어야 합니다.");
  }

  return x + y;
}
```

기본값이 필요하면 단축 평가 또는 ES6 기본 매개변수를 사용할 수 있다.

```js
function add(a = 0, b = 0, c = 0) {
  return a + b + c;
}

console.log(add(1, 2, 3)); // 6
console.log(add(1, 2));    // 3
console.log(add(1));       // 1
console.log(add());        // 0
```

### 12.5.3 매개변수의 최대 개수

매개변수는 적을수록 좋다. 매개변수가 많으면:

- 호출 시 순서를 기억하기 어렵다.
- 실수 가능성이 커진다.
- 함수가 여러 일을 하고 있을 가능성이 높다.
- 유지보수가 어려워진다.

권장 기준:

- 이상적인 매개변수 개수는 0개
- 가능하면 3개 이하
- 많아지면 객체 하나로 묶어 전달

```js
$.ajax({
  method: "POST",
  url: "/user",
  data: { id: 1, name: "Lee" },
  cache: false
});
```

객체를 인수로 전달하면 순서를 신경 쓰지 않아도 되고 의미가 명확해진다. 단, 함수 내부에서 객체를 변경하면 외부 객체도 변경되는 부수 효과가 생길 수 있다.

### 12.5.4 반환문

함수는 `return` 문으로 실행 결과를 반환할 수 있다.

```js
function multiply(x, y) {
  return x * y;
}

var result = multiply(3, 5);
console.log(result); // 15
```

반환문의 역할:

- 함수 실행을 중단하고 함수 몸체를 빠져나간다.
- `return` 뒤의 표현식을 평가해 반환한다.

```js
function multiply(x, y) {
  return x * y;
  // 이 아래 코드는 실행되지 않는다.
}
```

반환값을 생략하거나 반환문 자체를 생략하면 `undefined`가 반환된다.

```js
function foo() {
  return;
}

console.log(foo()); // undefined

function bar() {}

console.log(bar()); // undefined
```

`return` 키워드와 반환값 사이에 줄바꿈을 넣으면 ASI 때문에 의도치 않게 `undefined`가 반환될 수 있다.

```js
function multiply(x, y) {
  return
  x * y;
}

console.log(multiply(3, 5)); // undefined
```

## 12.6 참조에 의한 전달과 외부 상태의 변경

매개변수도 변수처럼 동작하므로 원시 값과 객체 전달 방식의 차이가 그대로 적용된다.

```js
function changeVal(primitive, obj) {
  primitive += 100;
  obj.name = "Kim";
}

var num = 100;
var person = { name: "Lee" };

changeVal(num, person);

console.log(num);    // 100
console.log(person); // { name: "Kim" }
```

- 원시 값: 값이 복사되어 전달되므로 함수 내부 변경이 원본에 영향을 주지 않는다.
- 객체: 참조 값이 복사되어 전달되므로 함수 내부에서 객체를 변경하면 원본도 변경된다.

외부 상태를 변경하는 함수는 상태 변화를 추적하기 어렵게 만든다. 이런 부수 효과를 줄이려면:

- 객체를 직접 변경하지 않는다.
- 필요한 경우 복사본을 만들어 변경한다.
- 순수 함수를 지향한다.

## 12.7 다양한 함수의 형태

### 12.7.1 즉시 실행 함수

함수 정의와 동시에 호출되는 함수를 즉시 실행 함수(IIFE)라고 한다. 단 한 번만 호출되며 다시 호출할 수 없다.

```js
(function () {
  var a = 3;
  var b = 5;
  return a * b;
})();
```

즉시 실행 함수는 보통 그룹 연산자 `()`로 감싼다. 함수 리터럴을 먼저 평가해 함수 객체를 만들기 위해서다.

즉시 실행 함수도 값을 반환하거나 인수를 전달받을 수 있다.

```js
var result = (function (a, b) {
  return a * b;
})(3, 5);

console.log(result); // 15
```

즉시 실행 함수는 변수나 함수 이름의 충돌을 피하는 데 유용하다.

### 12.7.2 재귀 함수

재귀 함수는 자기 자신을 호출하는 함수다. 반복되는 처리를 반복문 없이 표현할 수 있다.

```js
function countdown(n) {
  if (n < 0) return;
  console.log(n);
  countdown(n - 1);
}

countdown(10);
```

팩토리얼 예제:

```js
function factorial(n) {
  if (n <= 1) return 1;
  return n * factorial(n - 1);
}

console.log(factorial(5)); // 120
```

재귀 함수에는 반드시 탈출 조건이 필요하다. 탈출 조건이 없으면 무한 호출되어 스택 오버플로가 발생할 수 있다.

### 12.7.3 중첩 함수

함수 내부에 정의된 함수를 중첩 함수 또는 내부 함수라고 한다. 중첩 함수를 포함하는 함수는 외부 함수라고 한다.

```js
function outer() {
  var x = 1;

  function inner() {
    var y = 2;
    console.log(x + y);
  }

  inner();
}

outer(); // 3
```

중첩 함수는 보통 외부 함수를 돕는 헬퍼 함수 역할을 한다.

### 12.7.4 콜백 함수

함수의 매개변수를 통해 다른 함수 내부로 전달되는 함수를 콜백 함수라고 한다. 콜백 함수를 전달받는 함수를 고차 함수라고 한다.

```js
function repeat(n, f) {
  for (var i = 0; i < n; i++) {
    f(i);
  }
}

var logAll = function (i) {
  console.log(i);
};

repeat(5, logAll);
```

콜백을 사용하면 변하지 않는 반복 로직과 바뀌는 동작을 분리할 수 있다.

```js
repeat(5, function (i) {
  if (i % 2) console.log(i);
});
```

콜백 함수는 다음에서 자주 사용된다.

- 이벤트 처리
- 비동기 처리
- 배열 고차 함수

```js
setTimeout(function () {
  console.log("1초 경과");
}, 1000);

var doubled = [1, 2, 3].map(function (item) {
  return item * 2;
});

console.log(doubled); // [2, 4, 6]
```

### 12.7.5 순수 함수와 비순수 함수

순수 함수는 외부 상태에 의존하지 않고, 외부 상태를 변경하지 않는 함수다. 동일한 인수를 전달하면 언제나 동일한 값을 반환한다.

```js
var count = 0;

function increase(n) {
  return ++n;
}

count = increase(count);
console.log(count); // 1

count = increase(count);
console.log(count); // 2
```

비순수 함수는 외부 상태에 의존하거나 외부 상태를 변경하는 함수다.

```js
var count = 0;

function increase() {
  return ++count;
}

increase();
console.log(count); // 1

increase();
console.log(count); // 2
```

비순수 함수는 상태 변화를 추적하기 어렵게 만들 수 있다. 함수형 프로그래밍은 순수 함수와 보조 함수 조합을 통해 부수 효과를 줄이고 안정성을 높이려는 프로그래밍 방식이다.

---

## 내용 정리

- 함수는 일련의 과정을 코드 블록으로 묶은 실행 단위다.
- 함수는 코드 재사용, 중복 제거, 유지보수성, 가독성을 높인다.
- 함수는 객체이며 값처럼 다룰 수 있는 일급 객체다.
- 함수 선언문은 함수 호이스팅이 발생하고, 함수 표현식은 변수 호이스팅만 발생한다.
- 매개변수는 함수 내부에서만 유효하며, 인수 개수와 타입은 자동으로 검사되지 않는다.
- `return`은 함수 실행을 종료하고 값을 반환한다.
- 객체를 인수로 전달해 내부에서 변경하면 외부 객체도 변경되는 부수 효과가 생긴다.
- 콜백 함수와 고차 함수는 로직을 유연하게 합성하는 데 사용된다.
- 순수 함수는 외부 상태를 변경하지 않아 예측 가능성이 높다.

## 의문점 제기

1. 함수 선언문과 함수 표현식 중 팀 컨벤션으로 무엇을 기본으로 삼는 것이 좋을까?
2. 함수 호이스팅은 편리함보다 혼란을 더 많이 만들까?
3. 매개변수 타입 검사를 런타임 코드로 할지 TypeScript로 해결할지 기준은 무엇일까?
4. 매개변수가 많아질 때 객체 인수로 묶는 방식은 언제부터 적용하는 것이 좋을까?
5. 객체 인수를 함수 내부에서 변경하지 않도록 강제하려면 어떤 패턴이나 도구가 필요할까?
6. 콜백 함수가 많아질 때 가독성을 유지하려면 익명 함수와 기명 함수 중 무엇을 선택해야 할까?
7. 재귀 함수는 반복문보다 언제 더 직관적이며, 스택 오버플로 위험은 어떻게 줄일 수 있을까?
8. 순수 함수 중심 설계가 실제 프런트엔드 상태 관리에서 어떤 장점을 제공할까?

## 답변

1. **함수 선언문 vs 함수 표현식: 팀 기본 컨벤션**
   - **기본값으로 무난한 선택**은 “**함수 표현식(또는 화살표 함수)**을 우선”이다. 선언/사용 순서를 코드 흐름에 맞추기 쉬워서 읽는 사람이 예측하기 좋다.
   - **함수 선언문**은 “유틸/헬퍼처럼 파일 전역에서 어디서든 호출되는 성격”이거나, 선언 자체가 모듈의 중요한 API처럼 보이길 원할 때 잘 맞는다.
   - 팀 기준으로는 “**호이스팅에 기대지 않는다**”를 목표로 두고, 선언문을 쓰더라도 **사용보다 위에 배치**하는 규칙을 같이 두면 혼란이 줄어든다.

2. **함수 호이스팅: 편리함 vs 혼란**
   - 편리한 점: 선언 위치를 크게 신경 쓰지 않고 호출할 수 있다(특히 작은 스크립트/학습 코드).
   - 혼란스러운 점: “코드가 위에서 아래로 실행된다”는 직관과 어긋나서, 큰 코드베이스에서는 **정의 위치 탐색 비용**과 **리팩터링 위험**이 커진다.
   - 실무 결론: 호이스팅 자체가 나쁜 게 아니라, **호이스팅을 전제로 한 설계**가 커질수록 유지보수를 어렵게 만든다.

3. **타입 검사: 런타임 vs TypeScript 선택 기준**
   - **TypeScript로 충분한 경우**: 호출자가 같은 코드베이스(프런트엔드/모노레포 내부)이고, 빌드 단계에서 타입 안정성을 확보할 수 있을 때.
   - **런타임 검사까지 필요한 경우**:
     - 외부 입력(서버 응답, URL 파라미터, 사용자 입력, localStorage 등)을 받는다.
     - TS 타입이 “컴파일 타임 약속”일 뿐 실제 값은 깨질 수 있는 경로가 있다.
     - 라이브러리/공개 API처럼 “타입이 없는 호출자”도 고려해야 한다.
   - 요약: **외부 세계와 만나는 경계에서는 런타임 검증**, 내부 로직에서는 TS로 비용을 줄이는 식으로 나누는 게 일반적이다.

4. **매개변수가 많아질 때 객체 인수로 묶는 기준**
   - 단순히 “3개 넘으면 무조건 객체”라기보다 다음 신호가 오면 묶는 게 이득이다.
     - 같은 타입의 인수가 연달아 있어 **순서 실수**가 잦다.
     - 선택적 인수/기본값이 늘어 **호출부가 읽기 어려워진다**.
     - 함수가 커져서 옵션이 계속 추가될 전망이다.
   - 실무 팁: 객체로 묶을 때는 “필수/선택”을 명확히 하고(기본값/구조 분해), 옵션 이름이 곧 문서가 되도록 설계한다.

5. **객체 인수를 함수 내부에서 변경하지 않게 “강제”하는 방법**
   - **설계 패턴**: 입력을 “읽기 전용”으로 취급하고, 결과는 **새 객체로 반환**(불변 업데이트)한다.
   - **도구/규칙**:
     - 린터로 파라미터 재할당/변경을 제한하는 규칙을 둔다(예: 파라미터 변경 금지 성격).
     - TypeScript라면 입력 타입을 `readonly`로 모델링해 의도를 드러낸다.
   - 실무에서는 “함수 경계에서 복사본을 만들고 내부는 복사본만 수정” 같은 규칙도 흔하다(비용 vs 안전의 절충).

6. **콜백이 많아질 때: 익명 vs 기명 선택**
   - **짧고 지역적인 로직**(1~3줄, 의미가 명확): 익명 콜백이 간결하다.
   - **길어지거나 재사용/테스트가 필요**하거나, 스택 트레이스/로그에서 이름이 중요해지는 순간: **기명 함수로 분리**가 가독성과 디버깅에 유리하다.
   - 기준은 “콜백이 무엇을 하는지 한 번에 말로 설명 가능하냐”로 잡으면 좋다. 설명이 길어지면 분리할 타이밍이다.

7. **재귀가 더 직관적인 때와 스택 오버플로 줄이기**
   - 재귀가 직관적인 경우: **트리/그래프/중첩 구조**(DOM 탐색, 폴더 구조, 중첩 JSON)처럼 “부분 문제 = 전체 문제” 형태가 자연스러울 때.
   - 위험 줄이기:
     - **탈출 조건**을 명확히 하고, 입력 크기를 제한한다.
     - 깊이가 매우 깊을 수 있으면 **반복문 + 스택(배열)**로 바꾸는 방식이 안전하다.
     - 필요하면 작업을 쪼개 비동기로 양보하는 전략도 고려한다(긴 작업 블로킹 방지).

8. **순수 함수 중심 설계가 프런트엔드 상태 관리에서 주는 장점**
   - **예측 가능성**: 같은 입력(이전 상태, 액션)이면 같은 출력(다음 상태)이므로 버그 재현이 쉬워진다.
   - **변경 추적/최적화**: 불변 업데이트를 전제로 하면 참조 비교로 변경 여부를 빠르게 판단할 수 있어 렌더 최적화에 유리하다.
   - **디버깅/테스트**: 사이드 이펙트가 분리되면 단위 테스트가 쉬워지고, 상태 변화 흐름을 추적하기 좋다.
